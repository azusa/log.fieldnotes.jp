## 現状わかっていること：NVD方針転換と脆弱性管理の変化

### 1. 今回の変更の事実（確定事項）

2026年4月15日より、NISTはNVDのエンリッチメントをリスクベースの優先順位付けに切り替えた。優先対象はKEV掲載CVE・連邦政府使用ソフト・大統領令14028定義のクリティカルソフトウェアの3カテゴリのみで、それ以外は「Not Scheduled」として即時エンリッチされない。

NISTは独自のCVSSスコア付与も原則廃止し、CNA（ベンダー等の採番機関）が付与したスコアをそのまま採用する方針に変わった。

2026年3月1日以前の未エンリッチCVE約29,000件は一括でNot Scheduledに移行済み。

---

### 2. ツール・データへの影響（確認済み）

**Dependabot・SCAツール**
- DependabotはNVD直接ではなくGitHub Advisory Databaseを参照
- アラートが出るのはReviewed advisoryのみ
- NVDでCPEが付与されないCVEはReviewedへの昇格が遅れ、検出漏れリスクが増加

**CVSSスコア**
- DependabotのスコアはもともとCNA由来で、NISTスコアへの依存はない
- ただしCNAスコアとNISTスコアは半数超で乖離があり、品質のばらつきが拡大

**「Not Scheduled」の意味**
- NISTのリソース制約を示すものであり、低リスクを意味しない

---

### 3. 今後の方向性（方針は出たが未確定）

NISTはエンリッチメント作業をCNAに段階的に移管する方針を示しているが、そのためのガイダンスはまだ書かれておらず、移行スケジュールは未確定。

CISAのVulnrichmentプロジェクトとNISTの取り組みに重複があることが指摘されており、両機関の役割分担の調整が進行中。

EUは独自のEUVD（European Vulnerability Database）を2025年5月に立ち上げており、NVDと並走する形で動いている。ただし現状はNVDデータへの依存が大きく、完全な代替にはなっていない。

### 4. 実務として現時点で取れる対応（確立済み）

| 対応 | 根拠 |
|---|---|
| CISA KEVを一次トリアージソースにする | 実際の悪用確認済みという意味で最も信頼性が高い |
| EPSSを併用して悪用確率で優先度判断 | CVSSスコア単独判断の補完 |
| ベンダーアドバイザリを直接購読 | NVD経由の遅延・漏れを補完 |

| SLA・規程文書の「NVDスコアを基準とする」表現を修正 | スコアが存在しない・CNA由来で変動するという新前提への対応 |

## GitHub Advisory Database / Dependabotへの影響考察

### ひろの見立ての確認

GitHub Advisory DatabaseのReviewed setは、GitHubがサポートするパッケージエコシステム（npm・pip・Maven・Go等）を通じて配布されるソフトウェアに限定されている。これはReviewed advisoryがDependabotアラートを動かす仕組みと直結しているため、パッケージマネージャー上での依存関係にマッピングできないものはReviewedにならない。

つまりひろが言った通り、**npm・pip・Maven・Go等の主要エコシステムではパッケージ開発元がadvisoryを出せばReviewedに昇格するルートがある**ので、「NVDエンリッチ待ちでUnreviewedに滞留してDependabotが鳴らない」というリスクは相対的に小さい。

---

### それでも残る影響

**① CVSSスコアの品質ばらつき拡大（これが一番実務的に痛い）**

GitHubはCVSSスコアをNVDではなくパッケージメンテナー・CNA由来のものをそのまま採用している。NISTの検証がなくなることで、**ベンダーが自社製品の脆弱性スコアを低めに申告しても誰もチェックしない**状態になる。

DependabotのAuto-triage rulesでCVSSスコアの閾値を使って自動dismissしている場合、この品質劣化が静かに効いてくる。

**② NVD単独ルートのCVEは引き続きUnreviewed滞留リスクあり**

GitHubがReviewed advisoryを作成するソースはNVD・メンテナー申告・コミュニティ貢献の複数系統ある。

第三者研究者が発見してCVEとして申請したが**パッケージメンテナーがadvisoryを出していない**ケース、つまりメンテナーが把握していない・対応が遅い・OSS管理が手薄なパッケージは、NVDルートが詰まるとReviewedへの昇格が遅れる。

```
メンテナー自身がGHSA申告  → Reviewed昇格ルートあり（影響小）
第三者がCVE申請のみ       → NVD経由でUnreviewed → 今回の影響大
```

**③ パッケージ外ソフトウェアは元から死角（変化なしだが再確認が必要）**

Citrix NetScaler ADC等のプロプライエタリ製品はパッケージマネージャー経由で配布されないため、そもそもReviewedにならずDependabotアラートも出ない。

SIerとして顧客環境に入っているミドルウェア・インフラ製品の脆弱性はDependabotの検知対象外であることを改めて明示しておく必要がある。

**④ Dependabot Auto-triage rulesへの間接影響**

GitHubのAuto-triage rulesはseverity（CVSS由来）・EPSSスコア・scope（dev/prod）等でアラートを自動dismiss・スヌーズできる。CVSSスコアの信頼性が落ちると、このルールセットの前提が崩れる。**「severity: low は自動クローズ」みたいなルールを運用している場合は要見直し。**

---

### まとめると

| リスク | 影響度 | 理由 |
|---|---|---|
| 主要エコシステムのDependabot検出漏れ | **低** | メンテナーのGHSA申告ルートが生きている |
| 第三者発見CVEのUnreviewed滞留 | **中** | NVDルート詰まりの影響を受ける |
| CVSSスコア品質の劣化 | **中〜高** | NISによるベンダースコアの検証がなくなる |
| Auto-triage rulesの誤dismiss | **中** | スコア劣化が閾値ルールの信頼性を下げる |
| インフラ・ミドルウェア製品の死角 | **既存問題** | 今回の変化ではなく元からの限界 |

Dependabotを使ってる分には「即壊れる」わけじゃないけど、**スコアを信じすぎたルール設定と、メンテナーが不活発なOSSライブラリの組み合わせ**が一番危ない、というのが今回の考察の結論だよ😤

*【セキュリティ情報】脆弱性データベースの運用方針転換について* 

*■ まず前提：CVSSスコアとは何か*
ソフトウェアに脆弱性が発見されると、世界共通の識別番号「CVE」が付与されます。ニュースで「○○に致命的な脆弱性」と報道される「致命的」の根拠は、CVSS スコア（0〜10点）の区分に基づいています。
```


```
0.0        → None
0.1 〜 3.9 → Low
4.0 〜 6.9 → Medium
7.0 〜 8.9 → High
9.0 〜10.0 → Critical（致命的）
```

```
:bulb: *ちなみに：* 報道では最高レベルを「致命的な脆弱性」と表現することが多いですが、IPA（情報処理推進機構）の公式解説では Critical の日本語訳は *「緊急」* です（High＝重要、Medium＝警告、Low＝注意）。「致命的」は正式な訳語ではありませんが、業界用語として広く定着しています。

*CVE番号だけでは深刻度・影響製品は分かりません。* これに CVSS スコアや影響製品リスト（CPE）を付加する「エンリッチメント」作業を一手に担ってきたのが、脆弱性の分析・評価を手掛ける米国立標準技術研究所（NIST）が運営する NVD（National Vulnerability Database）です。

━━━━━━━━━━━━━━━━━━━━━━━━━
*■ 今回の変更：NVD エンリッチメントの大幅縮小（2026年4月15日施行）*

CVE 申請件数が 2020〜2025 年で *263% 増加* し、2025 年だけで約 49,000 件が登録されました。NIST は過去最多となる 42,000 件をエンリッチしましたが、増加ペースに追いつけず、方針を転換しました。

今後エンリッチメントの対象となるのは以下のみです。

• *CISA KEV カタログ掲載 CVE*（実際の悪用が確認済み） → 受領後1営業日以内
• 米連邦政府が使用するソフトウェアに影響する CVE
• 大統領令 14028 が定めるクリティカルソフトウェアに影響する CVE

上記以外の CVE は `Not Scheduled` ステータスとなり、CVSS スコアや影響製品情報が自動では付与されなくなります。全 CVE の *約 80%* が対象外となる見込みです。

━━━━━━━━━━━━━━━━━━━━━━━━━
*■ 私たちの業務への影響：2パターンで整理*

*【パターンA】npm・pip・Maven・Go 等の主要パッケージエコシステムを使う開発プロジェクト*

Dependabot は NVD を直接参照しておらず、*GitHub Advisory Database* を情報源としています。このデータベースでは、パッケージのメンテナー（開発元）が直接 advisory を申告するルートがあり、申告されたものは GitHub のレビューを経て Reviewed advisory となります。*Dependabot がアラートを出すのは Reviewed のみ* のため、メンテナー由来の情報が揃っている主要エコシステムでは検出漏れリスクは相対的に小さいです。

ただし以下の点には注意が必要です。

:warning: *CVSSスコアの品質ばらつき*
Dependabot が表示するスコアはメンテナー・ベンダー（CNA）が付与したもので、NIST による第三者検証がなくなります。ベンダーが自社製品の深刻度を低めに申告しても誰もチェックしない状態になるため、スコアの鵜呑みは禁物です。

:warning: *第三者発見CVEのUnreviewed滞留リスク*
パッケージメンテナーが把握・対応できていない脆弱性（第三者研究者がCVEを申請したケース等）は、NVD 経由の Unreviewed に滞留しやすくなります。管理が手薄なOSSライブラリを使っているプロジェクトは特に注意してください。

:warning: *Dependabot Auto-triage rules の見直し*
「severity: low は自動クローズ」等の CVSS スコア閾値ベースのルールを設定している場合、スコアの信頼性低下により意図しない自動 dismiss が発生するリスクがあります。ルール設定を見直してください。

├─ *【脚注】メンテナーの直接 advisory 申告ルートがあるエコシステム（GitHub Advisory Database 公式ドキュメントより）*
└ Composer / Erlang / Go / GitHub Actions / Maven / npm / NuGet / Pip / Pub / RubyGems / Rust / Swift
_上記以外のエコシステム（Docker・Terraform等）は NVD ルートのみとなり、Reviewed への昇格が遅れる可能性があります。_

---

*【パターンB】ミドルウェア・インフラ・プロプライエタリ製品を含む顧客環境*

Citrix・F5・VMware 等のプロプライエタリ製品や、パッケージマネージャー経由で配布されないソフトウェアは、*そもそも GitHub Advisory Database の Reviewed 対象外* です。Dependabot はこれらの脆弱性をアラートとして検出できません。これは今回の変更で新たに生じた問題ではなく、元からの構造的な限界ですが、NVD のエンリッチが縮小することで見落としリスクがさらに高まります。

:warning: *「Dependabot が鳴っていない＝安全」は成立しない*
インフラ・ミドルウェア製品については、Dependabot に頼らず CISA KEV カタログとベンダーアドバイザリを直接確認する体制が必要です。

:warning: *顧客向けセキュリティ報告書への影響*
NVD スコアを一次情報として引用している場合、スコア未付与の CVE が報告から漏れる可能性があります。特にインフラ製品に関する脆弱性の記載は情報源を見直してください。

━━━━━━━━━━━━━━━━━━━━━━━━━
*■ 現時点での対応として考えられること*

脆弱性管理の枠組み自体が過渡期にあり、業界全体として最適解が固まっていない状況です。現段階では以下が有効と考えられます。

1. *CISA KEV カタログ* を脆弱性トリアージの一次ソースとして参照する（パターンA・B 共通）
   → https://www.cisa.gov/known-exploited-vulnerabilities-catalog
2. CVSS スコアのみによる機械的なフィルタリングを廃止または補完し、*EPSS スコア*（悪用確率）との併用を検討する
3. 利用 OSS のベンダーアドバイザリ（npm security advisories 等）を Dependabot に依存せず直接購読する
4. インフラ・ミドルウェア製品については各ベンダーのセキュリティアドバイザリを個別に監視する体制を整える
5. 顧客向けの脆弱性管理規程・報告テンプレートに `Not Scheduled = 低リスク` と読める表現がないか確認する

:information_source: *`Not Scheduled` は NIST のリソース制約を示すものであり、その CVE が低リスクであることを意味しません。*
