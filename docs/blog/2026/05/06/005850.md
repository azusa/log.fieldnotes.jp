# 脆弱性情報に関するNISTの方針転換の方針に関する社内へのSlack案内テンプレ

※ AI生成です

```
*【セキュリティ情報】脆弱性データベースの運用方針転換について* 

*■ まず前提：CVSSスコアとは何か*
ソフトウェアに脆弱性が発見されると、世界共通の識別番号「CVE」が付与されます。ニュースで「〇〇に致命的な脆弱性」と報道される「致命的」(Critical)の根拠は、CVSS スコア（0〜10点）の区分に基づいています。
```

```
0.0        → None(なし)
0.1 〜 3.9 → Low(注意)
4.0 〜 6.9 → Medium(警告)
7.0 〜 8.9 → High(重要)
9.0 〜10.0 → Critical（緊急）
```

```

:bulb: *ちなみに：* 報道では最高レベルを「致命的な脆弱性」と表現することが多いですが、IPA（情報処理推進機構）の解説では Critical の日本語訳は *「緊急」* です（High＝重要、Medium＝警告、Low＝注意）。「致命的」は正式な訳語ではありませんが、業界用語として広く定着しています。

このスコアは「ネットワーク越しに攻撃できるか」「認証なしで悪用できるか」「機密性・完全性・可用性への影響はどの程度か」などのメトリクスを組み合わせて算出されます。

*CVE番号だけでは深刻度・影響製品は分かりません。* これに CVSS スコアや影響製品リスト（CPE）を付加する「エンリッチメント」作業を一手に担ってきたのが、脆弱性の分析・評価を手掛ける米国立標準技術研究所（NIST）が運営する NVD（National Vulnerability Database）です。Dependabot などのスキャナーも、この NVD のデータを前提に動作しています。

─────────────────────────
*■ 今回の変更：NVD エンリッチメントの大幅縮小（2026年4月15日施行）*

CVE 申請件数が 2020〜2025 年で *263% 増加* し、2025 年だけで約 49,000 件が登録されました。NIST は過去最多となる 42,000 件をエンリッチしましたが、増加ペースに追いつけず、方針を転換しました。

今後エンリッチメントの対象となるのは以下のみです。

• *CISA KEV カタログ掲載 CVE*（実際の悪用が確認済み） → 受領後1営業日以内
• 米連邦政府が使用するソフトウェアに影響する CVE
• 大統領令 14028 が定めるクリティカルソフトウェアに影響する CVE

上記以外の CVE は `Not Scheduled` ステータスとなり、CVSS スコアや影響製品情報が自動では付与されなくなります。全 CVE の *約 80%* が対象外となる見込みです。

─────────────────────────
:warning: GitHub Dependabot を使っている場合の注意点*

Dependabot は NVD を直接見ているわけではなく、*GitHub Advisory Database* を情報源としています。このデータベースには「Reviewed（レビュー済み）」と「Unreviewed（未レビュー）」の2種類があり、*Dependabot がアラートを出すのは Reviewed のみ* です。

*CVSSスコアの出どころについて*
Dependabot が表示する CVSS スコアは、NIST が独自に付けたスコアではなく、*ソフトウェアのメンテナーやベンダー（CNA）が付与したスコア* をそのまま採用しています。そのため NVD のエンリッチメント縮小が直接スコア表示に影響するわけではありません。

ただし以下の点に注意が必要です。

:warning: *「アラートが出ていない ≠ 安全」になるリスク*
NVD で CPE（影響製品の識別子）が付与されない CVE は、GitHub 側での Reviewed への昇格が遅れます。結果として Dependabot がアラートを出さないまま、実際には影響を受けるライブラリが放置されるケースが増える可能性があります。

:warning: *CVSSスコアの品質ばらつき*
CNA（ベンダー等）が付与するスコアと NIST が検証したスコアは、半数超の事例で乖離があります。「スコア7未満はスキップ」などの閾値ルールを機械的に適用している場合は見直しが必要です。



─────────────────────────
*■ 現時点での対応として考えられること*

脆弱性管理の枠組み自体が過渡期にあり、業界全体として最適解が固まっていない状況です。現段階では以下が有効と考えられます。

1. *CISA KEV カタログ* を脆弱性トリアージの一次ソースとして参照する
   → https://www.cisa.gov/known-exploited-vulnerabilities-catalog
2. CVSS スコアのみによる機械的なフィルタリングを廃止または補完し、*EPSS スコア*（悪用確率）との併用を検討する
3. 利用 OSS のベンダーアドバイザリ（npm security advisories 等）を Dependabot に依存せず直接購読する
4. 顧客向けの脆弱性管理規程・報告テンプレートに `Not Scheduled = 低リスク` と読める表現がないか確認する

:information_source: *`Not Scheduled` は NIST のリソース制約を示すものであり、その CVE が低リスクであることを意味しません。*

─────────────────────────
*■ 今後の動向*

NVD の方針転換を受けて、業界各社・標準化団体がどのように脆弱性管理の枠組みを再構築していくかは現時点では不透明です。引き続き NIST・CISA・各スキャナーベンダーの動向を注視し、対応方針のアップデートが必要な場合は改めて共有します。
```
