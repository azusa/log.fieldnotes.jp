## あず

わかる〜、それ典型的なやつ😂

「折り返してますが1行です」って注釈、最初に見たときは「まあ仕方ないか」って思うんだけど、長いコマンド何個も並ぶと注釈だらけになって、しかも読み手によっては「え、どこまでが1行？」って迷うんだよね。手順書のレビューで毎回つっこまれるやつ。

現実的な落とし所だと、こんな感じじゃない？

### 1. コードブロックは横スクロールさせる前提にする

Markdown系のレンダラーだと `<pre>` に `overflow-x: auto` 効かせれば横スクロールになるから、見た目の折り返しは諦めて「ブロック内は1行」というルールを徹底する。紙に出す前提じゃなければこれが一番ラク。

### 2. 長いオプションを環境変数や設定ファイルに逃がす

```
# 事前に変数定義
IMAGE=nginx:latest
NAME=myapp

docker run -d --name $NAME -p 8080:80 -e ENV=production $IMAGE
```

これだと本体のコマンドが短くなって1行に収まりやすい。手順書的にも「ここを変えればいい」が明確になるからレビューも通りやすい。

### 3. docker compose とか宣言ファイルに寄せる

長くなるってことはオプション多いってことだから、そもそもCLIで殴るのをやめて `compose.yaml` に書いて `docker compose up -d` で済ます。手順書も `vi compose.yaml` → 中身貼り付け → `compose up` の3ステップに分解できる。複数行ペースト禁止ルールとも相性いい。

ぶっちゃけ、JTCの手順書なら**2か3**が王道だと思うよ。1行コマンドを無理やり詰め込むより、「事前準備」と「実行」を分ける方が手順書としても堅い。

## あず

ハイハイ戻ります戻ります😂（聞いてないんかい）

そうそう、変数外出しで地味にハマるやつね。手順書通りにやったのに動かない、原因は引用符、っていうパターン本当に多い。

**典型的なやらかしパターン**いくつか挙げると：

**1. シングルクオートの中で展開されない**

```
NAME=myapp
echo 'Hello, $NAME'   # → Hello, $NAME （展開されない）
echo "Hello, $NAME"   # → Hello, myapp
```

これ、手順書で `'...'` をコピペした瞬間に「あれ？」ってなるやつ。書き手が気を抜くと普通に混在する。

**2. 環境変数とシェル変数の違い**

```
NAME=myapp           # シェル変数（子プロセスに継承されない）
export NAME=myapp    # 環境変数（継承される）
```

`docker run` の `-e` で渡すときとか、`sudo` で実行するときに「あれ、変数の中身空っぽじゃん」ってなるパターン。`sudo -E` 知らないと詰む。

**3. 変数代入の `=` の前後にスペース**

```
NAME = myapp   # ❌ コマンドとして解釈される
NAME=myapp     # ⭕
```

これ他言語から来た人が必ず一度はやる。

**4. 引用符なしで空白入りの値を入れる**

```
MSG=hello world           # ❌ "world" がコマンド扱い
MSG="hello world"         # ⭕
echo $MSG                 # 引用符なしだとIFSで分割される罠もあり
echo "$MSG"               # ⭕ 安全
```

**5. ヒアドキュメントで展開される/されない**

```
cat <<EOF      # 展開される
$NAME
EOF

cat <<'EOF'    # 展開されない（クオート付き）
$NAME
EOF
```

> 💡 ここらへんの事例は実機チェックはしてないけどあっている…はず…

これも手順書で「設定ファイルをcatで作る」とかやるときに罠になる。

…で、手順書の難しさってさ、**読者のシェルリテラシーをどこに置くか**問題があるよね。「変数に外出ししました、便利でしょ」って書いても、上のどれか1個でつまずいたら結局「コマンドそのまま貼ったほうがマシ」ってなっちゃう。

ぶう、結局**長い1行コマンドを折り返さずに載せる方法**に話が戻ってくるんだよね😂 ひろ、どっちで詰まってるの？「外出しの注意書きが長くなりすぎ」？それとも「外出ししても結局1行が長い」？

## あず

賢明な判断😂

シェルのクオートの話、入門レベルで「シングルは展開しない、ダブルはする」って書くと、ガチ勢から「`$()` と `` `...` `` の話は？」「`"$@"` と `"$*"` の違いは？」「`IFS` 変えたらどうなる？」「`set -u` した時の `${var:-}` は？」「POSIX shとbashで挙動違うとこあるよ？」って雨あられだからね🍺

> 💡 ここらへんの話をLLMに語らせて記事にしたらツッコミ多発で一大エンターテイメントになるのでは…

しかも厄介なのが、**ガチ勢のツッコミが全部正しい**ってとこ。間違ったこと言ってないから反論できない、けど読者の99%にはオーバースペック、っていう。

技術書・技術ブログの永遠の課題だよね：
- **正確に書く** → 入門者が脱落
- **わかりやすく書く** → ガチ勢が湧く
- **両方やる** → 分量が爆発して締切に間に合わない

…で、ひろみたいなベテランは「とりあえずこれ覚えとけ」の塩梅がうまいから本書けるんだろうけどさ、その塩梅の説明自体がまた長いんだよね😂
